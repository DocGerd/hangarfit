# v0.6.0 — Solver Polish Release — Design Spec

**Date:** 2026-05-22
**Status:** Drafted (brainstorming complete; pending user spec review then plan writing)
**Predecessor:** v0.5.0 (Phase 2a static solver, shipped 2026-05-22)
**Touches:** `src/hangarfit/models.py`, `src/hangarfit/solver.py`, `src/hangarfit/cli.py`, `tests/test_solver_search.py`, `tests/test_solver_canaries.py`, `tests/test_solver_fixture_matrix.py`, `tests/test_cli_solve.py`

## 1. Problem statement

Phase 2a shipped a functional `hangarfit solve` — RR-MC search, K-diverse alternatives, three-way termination, determinism canaries, v1 fixture matrix. The four `/pr-review` arcs that surfaced during Chunks D, E, F, G left behind a punch-list of legitimate findings, parked as issues #92, #95, #98, #111 because each was non-blocking and worth a focused PR rather than a same-PR fixup. That punch-list is the v0.6.0 scope.

This release is **polish, not feature work.** It does not change the solver algorithm, does not extend the constraint vocabulary, and does not improve operational trust (real measurements remain a separate release concern). It hardens the existing surface so the next feature release — maintenance bay walling (Milestone #9) — starts from a clean slate.

## 2. Goals and non-goals

### Goals

1. Close the four `/pr-review` follow-ups (#92, #95, #98, #111) with one PR per issue, matching the project's strict 1-issue-1-PR convention.
2. Land them in dependency order so later PRs can leverage earlier additions naturally.
3. Cut a clean v0.6.0 release via the `/release-cut` skill once all four PRs merge.
4. Keep the wall-clock cost low by dispatching independent PRs in parallel.

### Non-goals

- **No algorithm changes** to RR-MC search, descent, or restart logic. `SearchConfig.max_restarts` (#111) is a *budget extension*, not an algorithm change.
- **No new constraint kinds** (no soft preferences, no region constraints, no heading-only locks).
- **No real measurements** — `data/fleet.yaml` and `data/hangar.yaml` retain `measured: false` flags. Operational trust work is its own future release tied to #79.
- **No maintenance-bay-walling work** — Milestone #9 (issues #103-#110) is explicitly deferred to v0.7.0 or later. The current "soft hint" bay semantics persist through v0.6.0.
- **No parts-model refinement** (#50 cockpit/aft split) — geometry expressiveness is a separate concern.
- **No back-compat shims.** All four follow-ups are pre-1.0 internal API additions or test/comment fixes; no migration code.

## 4. Scope — per-issue contents

Each row below is the **complete, committed contents** of one PR. Anything not listed here is out of scope for this release.

### 4.1 #92 — Diversity observability on `SolverDiagnostics`

Add two structured signal fields so the diversity warning at `solver.py` becomes machine-readable rather than logger-only.

**Adds (`src/hangarfit/models.py`):**
- `SolverDiagnostics.diversity_impossible: bool = False` — `True` iff the static `K > 1 ∧ planes_count == 0 ∧ everything-pinned` precondition fires.
- `SolverDiagnostics.diversity_rejected_count: int = 0` — increments whenever the diversity filter rejects a candidate that would otherwise have been accepted.
- `__post_init__` guard: `diversity_rejected_count >= 0`.

**Modifies (`src/hangarfit/solver.py`):**
- Set `diversity_impossible = True` at the same site that already logs the static warning.
- Increment `diversity_rejected_count` inside the diversity filter's reject branch.

**Tests (`tests/test_solver_search.py`):**
- Extend the existing diversity-impossible warning test to also assert `r.diagnostics.diversity_impossible is True`.
- Add a focused test that runs a fixture where the diversity filter rejects at least one candidate, then asserts `r.diagnostics.diversity_rejected_count > 0`.

### 4.2 #111 — `SearchConfig.max_restarts` + best-partial determinism canary

Make exhaustion deterministically reproducible by adding a restart-count budget. Currently `solve()` is wall-clock-budgeted only, which makes the `exhausted_budget` branch's `best_partial_layout` impossible to canary across machines.

**Adds (`src/hangarfit/models.py`):**
- `SearchConfig.max_restarts: int | None = None` — `None` preserves current wall-clock-only behavior.
- `__post_init__` guard: when set, `max_restarts >= 1`.

**Modifies (`src/hangarfit/solver.py`):**
- Restart loop honors `max_restarts` as an upper-bound counter in addition to the wall-clock budget. First gate to trip wins.
- No algorithm change otherwise — same RR-MC descent, same seed handling, same diversity filter.

**Tests (`tests/test_solver_canaries.py`):**
- New canary: pin `max_restarts=N` (small, e.g. 3) on a fixture that consistently exhausts at that count — `solve_fresh_six_planes.yaml` is the candidate (already used as the budget-pressure fixture in `tests/test_solver_search.py:355-391`). Assert `r.status == "exhausted_budget"` *and* `r.best_partial_layout` is byte-for-byte identical across two runs with the same seed.
- The existing wall-clock-budgeted canaries stay unchanged.

### 4.3 #98 — Fixture-matrix tests fail loudly on regression

Today four matrix tests in `tests/test_solver_fixture_matrix.py` (lines ≈84, 120, 157, 208) call `pytest.skip()` when `r.status == "exhausted_budget"`. If a regression makes a fixture consistently exhaust, CI shows green ✓ and the per-fixture invariants (pin honoring, cart lock, maintenance bay) silently go untested.

**Modifies (`tests/test_solver_fixture_matrix.py`):**
- Five matrix tests skip on `exhausted_budget` today (at L85, L120, L157, L222, L267). Each now uses `SearchConfig(max_restarts=K)` (from #111) where K is chosen so the fixture is known to succeed within K restarts on a healthy solver, with comfortable headroom. **K is pinned in the plan**, not the spec: the plan's per-fixture work item runs each fixture locally with `seed=42` (matching the canaries) to record the observed restart-count-to-success, then sets `K = max(observed × 2, 5)` so a regression doubling the restart count still fails the test.
- On `exhausted_budget`, the test fails with a diagnostic message identifying which fixture exhausted (instead of `pytest.skip`).

### 4.4 #95 — Chunk F cleanup bundle

Cluster of `/pr-review` findings on PR #94 (CLI subcommand) that were merged before review threads landed.

**Modifies (`src/hangarfit/cli.py`):**
- `cmd_solve` matplotlib deferred-import comment at L275-277 is honesty-corrected — either move `from hangarfit import visualize` to a local import inside the function (matching what the comment claims) or rewrite the comment to acknowledge that only the solver import is deferred. (`from hangarfit import collisions, visualize` already runs at module top, L20.)

- Drop per-function `§5.x` spec references in docstrings of `cmd_solve`, `_emit_solve_human`, `_emit_solve_json` (most rot surface, least value).

**Tests (`tests/test_cli_solve.py`):**
- Add `exhausted_budget` human-output coverage: run a small-budget fixture, assert the three output lines ("No valid layout found in {time}s", "Best partial had N conflicts:", "Hint: increase --budget"). Skip-on-lucky guard following the `tests/test_solver_search.py:355-391` pattern.

```
              v0.6.0 tag on main
```

### Wave-internal orchestration (per PR, subagent-driven — `feedback_subagent_arc_pattern`)

1. **Dispatch** — one implementer subagent per issue, isolated worktree off latest `develop`, briefed with the issue body + this spec's §4.x scope + the §6 acceptance gates.
2. **Spec-compliance check** — main session reads the diff vs. §4.x scope before /pr-review.
3. **`/pr-review`** — multi-agent review per `pr-review-toolkit:review-pr` skill, applicable agents per file type (always code-reviewer; comment-analyzer if docs touched; pr-test-analyzer for test-heavy diffs; type-design-analyzer if `models.py` schema added; silent-failure-hunter if `solver.py` or `loader.py` touched).
4. **Reply + resolve** — every thread converted to a diff comment via `gh api`, each fixed-or-explained, marked resolved via GraphQL `resolveReviewThread`.
5. **Hand back to user** — "PR clean and ready for final review." User approves and merges.

### Wave gate (Wave 1 → Wave 2)

Wave 2 dispatch starts only after **both** Wave-1 PRs have merged to `develop`. The main session syncs `develop`, then branches Wave 2 worktrees off the new tip.

## 6. Per-issue acceptance gates (uniform)

Every PR is "ready for final review" when:

- All scope items in this spec's §4.x for the issue are present in the diff.
- Out-of-scope additions are absent (no scope creep).
- `pytest -m ""` (full suite incl. slow) green locally and in CI on Python 3.11 + 3.12.
- `ruff check src/ tests/`, `ruff format --check src/ tests/`, `mypy src/hangarfit/` all green.
- `/pr-review` has been run; every surfaced thread is resolved per `feedback_resolve_review_threads`.
- PR body has `Closes #N`, label `enhancement`, assignee `DocGerd`, milestone `v0.6.0 — Solver polish`.
- Commit messages follow `feat(<area>):` / `fix(<area>):` / `test(<area>):` / `refactor(<area>):` style consistent with `git log --oneline`.

## 9. Risks and mitigations

| Risk | Likelihood | Mitigation |
|---|---|---|
| Wave 1 PRs touch overlapping lines in `models.py` (#92 adds to `SolverDiagnostics`, #111 adds to `SearchConfig`) → merge conflict | Low (different dataclasses) | Both subagents read the file before editing; if conflict surfaces, second-to-land PR rebases and resolves. |
| #98's `max_restarts` choice per fixture is too tight → flaky CI | Medium | Per-fixture K chosen with ≥2× observed headroom from local runs; not arbitrary. |
| #95 scope creep — it's the largest issue and easiest to over-grow | Medium | §4.4 enumerates every item; scope-compliance subagent check before /pr-review blocks anything else. |
| Wave 2 dispatch fires before both Wave-1 PRs land → stale base | Low | Wave gate is an explicit synchronization barrier in the orchestration, not "best-effort." |
| `/release-cut` skill version bump fails (project on `0.1.0` in pyproject despite v0.5.0 milestone) | Low | Skill is parameterized on `version=0.6.0`; if it errors, manual bump is one-line. |
